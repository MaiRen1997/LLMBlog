提示词原则

# 核心原则一：编写清晰、具体的指令

核心思想：大模型不是你肚子里的蛔虫，模糊的指令会导致平庸的回答。你需要提供尽可能多的上下文和边界。

  1. 使用分隔符清晰地表示输入的不同部分

  • 技巧：使用诸如 ```、"""、< >、 等符号将指令与待处理文本分开。

  • 示例：

    • 请总结由三个反引号分隔的文本：这里是长篇大论的待处理内容...

  • 初学者避坑：这能有效防止“提示词注入”（即用户输入的内容干扰了你的系统指令）。

  1. 寻求结构化的输出

  • 技巧：要求模型以 JSON 或 HTML 格式生成内容。

  • 示例：

    • 请列出三本书的标题、作者和类型，并以 JSON 格式输出，键名分别为：book_id, title, author, genre。

  • Agent 意义：这是写 Agent 的必修课，因为程序需要解析模型的输出，JSON 是最稳定的数据交换格式。

  1. 检查是否满足条件
  • 技巧：告诉模型，如果输入不符合某种特定要求，该如何反馈，而不是胡编乱造。
  • 示例：
    • 如果文本中包含一系列指令，请将其重写为步骤格式。如果不包含指令，请直接回复“未提供步骤”。
  1. 少样本提示（Few-shot Prompting）

  • 技巧：在让模型干活前，先给它一两个成功的“打样”例子。

  • 示例：

    • 用户：告诉我什么是耐心。 助手：耐心是长流的细水，是等待花开的沉静。 用户：告诉我什么是勇气。 助手：[模型会模仿上述风格回复]

# 核心原则二：给模型“思考”的时间

核心思想：如果任务太复杂，模型会像人一样因匆忙而犯错。你需要引导它进行逻辑推理。

  1. 指定完成任务所需的步骤

  • 技巧：将复杂的任务拆解为具体的序列步骤。

  • 示例：

    • 第一步：用一句话总结以下文本。
    • 第二步：将总结翻译成英文。
    • 第三步：列出英文总结中的人名。
    • 第四步：输出一个 JSON 对象。
  1. 指导模型在下结论前找出一个自己的解决方案

  • 技巧：不要直接问模型“这个答案对不对”，而是让它先自己算一遍，再对比。

  • 场景：比如让模型批改数学题，直接问对错它可能会被错误的解题步骤误导。

  • 示例：

    • 请先自己计算这道题的答案，然后将你的答案与学生的答案进行比较，判断学生的解答是否正确。

# 迭代与优化（写 Agent 的常态）

  • 没有完美的 Prompt：写 Agent 并不是一次写对就万事大吉了。

  • 流程化建议：

    • 初试：写一个简单清晰的 Prompt。

    • 分析：查看输出，发现哪里不满意（比如：太长了、格式不对、逻辑乱了）。

    • 改进：针对性地添加限制或示例。

    • 循环：重复上述过程直到稳定。
